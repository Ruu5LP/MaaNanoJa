# CLAUDE.md — 開発ガイド（エージェント/開発者の入口）

このリポジトリで作業する前に、まずここを読む。**どこに・何を・どう書くか**と、**守るべきルール**を1枚にまとめてある。

> [!important] 受け入れ基準（この設計の目的）
> **「次にこのコードを触るAI（や人）が、一番理解しやすく・壊しにくい状態か」**。
> 迷ったらこの基準で判断する。賢い書き方より、**読んで即わかる・型とテストが守ってくれる**書き方を選ぶ。
> 誰が入っても同じルールで同じ品質を出せること。それがこのリポジトリの狙い。

## これは何か

月イチの家麻雀のスコアと、放銃・流局・立直などの局データを記録して分析する**ローカル完結**の Web アプリ。
バックエンドなし。データはブラウザの localStorage に持つ。振る舞いの仕様は [SPEC.md](./SPEC.md) を参照。

## 技術スタック

- React 18 + Vite + **TypeScript（strict）**
- テスト: Vitest / Lint: ESLint(flat) / 整形: Prettier
- 依存は最小限。UIライブラリ・状態管理ライブラリ・CSSフレームワークは入れない（増やすときは理由を書く）。

## ディレクトリと責務

```
src/
  lib/          純粋ロジック（副作用なし・ブラウザ非依存・テスト対象）
    domain.ts     型定義 = 仕様の中心。データ構造の意味と制約はここ。
    scoring.ts    スコア/点数計算（純粋関数）
    game.ts       半荘の進行・局ログの再生（純粋関数）
    stats.ts      成績集計（純粋関数）
    store.ts      localStorage 読み書き＋初期データ
    *.test.ts     lib のテスト（Vitest）
  views/        画面（1タブ1ファイル）。表示とユーザー操作に専念。
  App.tsx       状態(db)の保持と、更新アクション(api)の集約。
  main.tsx      エントリ
  styles.css    スタイル（CSS変数トークン）
```

### 設計ルール（必須）

- **ロジックは `lib/` に、純粋関数で書く。** `views/` に計算ロジックを持ち込まない。`lib/` は React も DOM も import しない。
- **状態は `App.tsx` の単一 `db` に集約し、更新は `api` 経由。** `views/` は `db`（読み）と `api`（書き）だけを受け取る。localStorage を直接触らない。
- **型で縛る。** ドメインは `domain.ts` の型で表現する。特に `Hand` は判別可能ユニオン。`any` は原則禁止（外部入力の境界だけ `unknown`→検証して絞る）。
- **数値の丸め・単位を勝手に変えない。** スコアは「÷1000」単位。既存スプレッドシートとの一致（[SPEC.md](./SPEC.md)）を壊さない。
- **スキーマ変更は移行付きで。** 保存データは後方互換で読む（`store.ts#normalizeDB`）。形を変えるなら `version` を上げ、移行を書く。
- **UI文言・コメントは日本語。** ファイル名は ASCII。

## 必ず通すチェック（CIでゲート化）

コミット前・PR前に必ず緑にする。CI（`.github/workflows/ci.yml`）でも同じものが回り、**緑でないとマージしない**。

```bash
npm run check   # typecheck → lint(警告0) → format:check → test をまとめて実行
```

個別:

```bash
npm run typecheck   # tsc --noEmit（strict）
npm run lint        # eslint（--max-warnings 0 相当。警告も残さない）
npm run format      # prettier --write（整形を当てる）
npm test            # vitest run
npm run build       # 本番ビルド（typecheck込み）
```

新しい振る舞いを足したら、**`lib/` のロジックにはテストを書く**（特にスコア/点数計算）。テストは「実行できる仕様」。

## スタイルの約束

- 色・余白は `styles.css` の CSS 変数（トークン）を使う。生の16進数を各所に散らさない。
- **着順の色は順序色（1位→4位: 青→赤）**。CVD（色覚多様性）検証済みの値を使い、必ず数値ラベルを併記する（色だけで意味を伝えない）。
- モバイル前提。数値は `font-variant-numeric: tabular-nums`。

## 作業の流儀

- 小さく作って、`npm run check` を緑にしてからコミット。コミットメッセージは日本語で簡潔に。
- **作り終わり＝記録とルールも残して終わり。** 大きめの設計判断をしたら、その理由を短く残す（このファイルや SPEC を更新する）。次に触る人が迷わないように。
- 構成やルールが実態に合わなくなったら、黙って従い続けず、より良い形を提案してから変える。
