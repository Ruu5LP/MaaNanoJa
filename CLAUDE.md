## これは何か

月イチの家麻雀のスコアと、放銃・流局・立直などの局データを記録して分析する**ローカル完結**の Web アプリ。
基本はバックエンドなし・データはブラウザの localStorage に持つ。振る舞いの仕様は [SPEC.md](./SPEC.md) を参照。

**LAN同期モード（任意）**: 家麻雀の当日、PCを親機にして「PCのモニターに全員ぶんを映しつつ、入力はスマホ・PCどちらからでも」を実現するための、同じWiFi内だけで完結する同期を後付けできる（`server/`）。外部クラウドには一切出さない＝"ローカル(LAN)完結"は保つ。サーバが居なければ従来どおり localStorage だけで動く。詳細は後述の「LAN同期モード」。

## 技術スタック

- React 18 + Vite + **TypeScript（strict）**
- テスト: Vitest / Lint: ESLint(flat) / 整形: Prettier
- 依存は最小限。UIライブラリ・状態管理ライブラリ・CSSフレームワークは入れない（増やすときは理由を書く）。

## ディレクトリと責務

```
src/
  lib/          純粋ロジック（副作用なし・ブラウザ非依存・テスト対象）
    domain.ts     型定義 = 仕様の中心。データ構造の意味と制約はここ。
    scoring.ts    スコア/点数計算（純粋関数）
    game.ts       半荘の進行・局ログの再生（純粋関数）
    stats.ts      成績集計（純粋関数）
    store.ts      localStorage 読み書き＋初期データ
    remote.ts     LAN同期サーバとの通信境界（確定データ=DBの純粋な判定関数＋fetch）
    live.ts       LAN同期の「実況」チャンネルの境界（入力中プレビュー用の純粋判定＋fetch）
    *.test.ts     lib のテスト（Vitest）
  views/        画面（1タブ1ファイル）。表示とユーザー操作に専念。
    LivePreview.tsx は例外的にタブではなく、他端末の入力中を映す読み取り専用オーバーレイ。
  App.tsx       状態(db)の保持と、更新アクション(api)の集約。
  useLanSync.ts  LAN同期のReactフック（サーバが居れば db をサーバと同期）
  useLiveInput.ts 入力中プレビューのReactフック（発信=usePublishLive / 受信=useWatchLive）
  main.tsx      エントリ
  styles.css    スタイル（CSS変数トークン）
server/
  server.mjs    LAN同期サーバ（依存ゼロ・Node標準のみ）。dist配信＋/api/db＋/api/live
  data/         実行時データ db.json（.gitignore。コミットしない）
```

### 設計ルール（必須）

- **ロジックは `lib/` に、純粋関数で書く。** `views/` に計算ロジックを持ち込まない。`lib/` は React も DOM も import しない。
- **状態は `App.tsx` の単一 `db` に集約し、更新は `api` 経由。** `views/` は `db`（読み）と `api`（書き）だけを受け取る。localStorage を直接触らない。
- **型で縛る。** ドメインは `domain.ts` の型で表現する。特に `Hand` は判別可能ユニオン。`any` は原則禁止（外部入力の境界だけ `unknown`→検証して絞る）。
- **数値の丸め・単位を勝手に変えない。** スコアは「÷1000」単位。既存スプレッドシートとの一致（[SPEC.md](./SPEC.md)）を壊さない。
- **スキーマ変更は移行付きで。** 保存データは後方互換で読む（`store.ts#normalizeDB`）。形を変えるなら `version` を上げ、移行を書く。
- **UI文言・コメントは日本語。** ファイル名は ASCII。

## 必ず通すチェック（CIでゲート化）

コミット前・PR前に必ず緑にする。CI（`.github/workflows/ci.yml`）でも同じものが回り、**緑でないとマージしない**。

```bash
npm run check   # typecheck → lint(警告0) → format:check → test をまとめて実行
```

個別:

```bash
npm run typecheck   # tsc --noEmit（strict）
npm run lint        # eslint（--max-warnings 0 相当。警告も残さない）
npm run format      # prettier --write（整形を当てる）
npm test            # vitest run
npm run build       # 本番ビルド（typecheck込み）
```

新しい振る舞いを足したら、**`lib/` のロジックにはテストを書く**（特にスコア/点数計算）。テストは「実行できる仕様」。

## スタイルの約束

- 色・余白は `styles.css` の CSS 変数（トークン）を使う。生の16進数を各所に散らさない。
- **着順の色は順序色（1位→4位: 青→赤）**。CVD（色覚多様性）検証済みの値を使い、必ず数値ラベルを併記する（色だけで意味を伝えない）。
- モバイル前提。数値は `font-variant-numeric: tabular-nums`。

## LAN同期モード（家麻雀の当日に使う任意機能）

「PCモニターに全員ぶんを映しつつ、入力はスマホ・PCどちらからでも」を、同じWiFi内だけで実現する仕組み。

**使い方（当日、PCで）:**

```bash
npm run host   # ビルド＋サーバ起動（これ1つ）。起動ログに各端末で開くURLが出る
```

起動後、PCもスマホも `http://<PCのローカルIP>:5180`（ログに表示される）を開くだけ。全端末が同じデータを見て・書ける。止めるときは `Ctrl+C`。データは `server/data/db.json`（＝PC上のファイル。外部に送らない）。

**仕組みと設計判断:**

- サーバ（`server/server.mjs`）は **依存ゼロ・Node標準のみ**。`dist/` を配信し、`GET/PUT /api/db` で共有DBを出し入れするだけ。「依存は最小限」の約束を守るため、あえてExpress等を入れていない。
- クライアントは起動時に `/api/db` を叩いて**サーバの有無を自動判定**する（`useLanSync.ts`）。居れば同期モード、居なければ従来どおり localStorage だけ。**普段の `npm run dev` は一切変わらない。**
- 同期は **DB全体＋版数(rev) の last-write-wins**。1秒ごとにサーバを見に行き、別端末の変更を取り込む。ロジックの分岐判定（取り込むべきか等）は `lib/remote.ts` の純粋関数に切り出してテスト済み。
- localStorage への保存は**端末ごとのバックアップ**として同期モードでも継続（サーバが落ちてもその端末のデータは残る）。

**入力中プレビュー（実況）:**

- 半荘は「半荘を終了→保存」するまで DB に入らない。そのため進行中の半荘は他端末からは見えない。これを補うのが**実況チャンネル**（`/api/live`）。「今まさに入力している半荘（スコアボード＋打っている最中の1局）」を他端末の画面上部にライブで映す。
- 実況はサーバの**メモリ上だけ**に持ち、`db.json` にもファイルにも残さない（消えていい一時状態＝プレゼンス）。落ちても記録には一切影響しない best-effort。
- **確定データ(DB)には混ぜない。** DBの last-write-wins 同期を汚さないよう、実況は別チャンネル（`lib/live.ts` / `useLiveInput.ts`）に分離している。
- 発信側（入力中の端末）は中身が変わるたび送信＋2秒ごとにハートビート。受信側は1秒ごとに見に行き、鮮度切れ（6秒）や自分の実況は映さない。鮮度は端末間の時計ズレを避けるため**サーバ時計**（応答の `ts`/`now`）で測る。判定は `lib/live.ts#shouldShowLive`（純粋関数・テスト済み）。
- 割り切り: 実況は「1人が入力し、他が見る」前提。同時に2端末が別々の半荘を入力すると、後から送った方だけが映る（実況もサーバ1枠の last-write-wins）。

> [!warning] LAN同期の割り切り（既知の限界）
> **完全に同じ瞬間**に別々の端末で別々の入力を確定すると、後勝ちで片方が消える（DB全体を丸ごと送る last-write-wins のため）。家麻雀は「1局ぶんを誰か1人が入力する」流れなので実質起きない前提で割り切っている。もし複数人が本当に同時入力する運用に変わったら、フィールド単位のマージ（CRDT等）が必要になる — そのときは設計から見直す。
> クラウド同期（家の外や別WiFiからの同期）は**やっていない**。あくまで同じWiFi内だけ。外に出さないことがプライバシー上の利点でもある。

## 作業の流儀

- 小さく作って、`npm run check` を緑にしてからコミット。コミットメッセージは日本語で簡潔に。
- **作り終わり＝記録とルールも残して終わり。** 大きめの設計判断をしたら、その理由を短く残す（このファイルや SPEC を更新する）。次に触る人が迷わないように。
- 構成やルールが実態に合わなくなったら、黙って従い続けず、より良い形を提案してから変える。

## 入力UIの受け入れ基準

> [!warning] 教訓（2026-07-22）
> 初版の局ログ入力は、型チェック・テストが緑でも「実際に家麻雀のその場で使うと操作しづらい」状態だった
> （プルダウンで翻符を選ぶ、ダブロン非対応、など）。**動くこと ≠ 使いやすいこと**。同じ失敗を繰り返さないための基準を明文化する。

入力系の画面（局ログ入力など）を作る／変更するときは、`npm run check` が緑になることに加えて、次を自問してから完成とする。

- **実際の操作シーン**を具体的に思い浮かべたか（家麻雀のその場、対局を止めずに片手〜両手で入力する場面）
- タップ/選択の**回数と種類**は現実的か（プルダウンでの長いリストからの選択は避け、ボタン・早見表など直接選べる形を優先）
- **複数人が同時に絡むケース**（ダブロン・トリプルロンなど、実際の対局で起きる分岐）を考慮したか
- 数値の**自動計算が間違っていた場合に、手動で補正できる**手段があるか（積み棒の手動修正のように）
- 迷ったら作り切ってから直すのではなく、早い段階でRuuに触ってもらって確認する

参考: [Ruu5LP/airuu-inc](https://github.com/Ruu5LP/airuu-inc) の決定ログ「麻雀トラッカーの局ログ入力をRuuの指摘で作り直した」（2026-07-22）
